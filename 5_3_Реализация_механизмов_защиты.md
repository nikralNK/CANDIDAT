# 5.3 Реализация механизмов защиты

В разрабатываемой информационной системе были реализованы базовые механизмы защиты, направленные на ограничение доступа к приложению, контроль корректности вводимых данных и защиту данных при работе с базой MS SQL Server.

Так как проект является учебным прототипом, реализованные механизмы не заменяют полноценную промышленную систему защиты, но показывают основные подходы, которые могут применяться в информационной системе учета кандидатов.

## Реализация авторизации пользователей

В приложении реализовано отдельное окно входа `LoginWindow`. Теперь при запуске сначала открывается форма авторизации, а главное окно приложения становится доступно только после успешного ввода логина и пароля.

Для учебной демонстрации в базе данных автоматически создаются два пользователя:

- `hr / hr123` — пользователь с ролью `Кадровик`;
- `admin / admin123` — пользователь с ролью `Администратор`.

Создание пользователей выполняется методом `EnsureDefaultUsers()` в классе `DatabaseService`. Это сделано для того, чтобы после запуска приложения сразу можно было проверить работу механизма входа.

После успешного входа данные пользователя передаются в главное окно приложения через объект `UserSession`. В верхней части главного окна отображается текущий пользователь и его роль.

## Хэширование паролей

Пароли пользователей не сохраняются в базе данных в открытом виде. Перед сохранением и проверкой пароль преобразуется в хэш SHA-256.

Для этого в классе `DatabaseService` реализован метод `HashPassword()`. Он получает строку пароля, переводит ее в массив байтов и вычисляет хэш с помощью `SHA256`.

Использование хэширования позволяет не хранить пароль обычным текстом. При входе в систему введенный пароль снова хэшируется, после чего полученный хэш сравнивается со значением в базе данных.

## Проверка логина и пароля

Проверка пользователя выполняется методом `AuthenticateUser()`. Метод получает логин и пароль, хэширует пароль и выполняет запрос к таблице `Users`.

Если пользователь с указанным логином и хэшем пароля найден, метод возвращает объект `UserSession`. Если данные неверные, возвращается `null`, а приложение показывает сообщение `Неверный логин или пароль`.

Таким образом, пользователь не может перейти к основной форме приложения без успешной авторизации.

## Проверка подключения к базе данных

Перед авторизацией выполняется проверка подключения к базе данных. Для этого используется метод `CanConnect()`. Если приложение не может подключиться к MS SQL Server, пользователь получает сообщение об ошибке.

Это нужно для того, чтобы приложение не продолжало работу без доступа к таблицам пользователей и основным данным системы.

## Проверка вводимых данных

В главной форме приложения реализована проверка данных перед сохранением карточки кандидата. Проверки находятся в методе `ValidateCardForm()`.

Проверяются следующие данные:

- фамилия кандидата;
- имя кандидата;
- отчество кандидата;
- дата рождения;
- возраст кандидата не меньше 18 лет;
- уровень образования;
- источник сведений;
- должность;
- наименование службы;
- дата заявления.

Если обязательное поле не заполнено, приложение не сохраняет карточку и выводит предупреждение.

## Ограничение даты рождения

Для даты рождения кандидата реализовано ограничение возраста. Метод `GetMaxCandidateBirthDate()` возвращает дату, соответствующую возрасту 18 лет.

В календаре `BirthDatePicker` запрещается выбирать дату рождения позже этой даты. Также проверка выполняется при сохранении карточки, поэтому даже при ручном вводе некорректная дата не будет сохранена.

## Использование справочников

Для уменьшения ошибок ручного ввода в приложении используются выпадающие списки. Кадровик выбирает значения из справочников, а не вводит их вручную.

Реализованы следующие справочники:

- источник сведений;
- категория службы;
- должность;
- наименование службы;
- тип этапа изучения;
- тип служебной проверки.

После выбора категории службы приложение показывает только подходящие должности. После выбора должности отображаются связанные с ней службы. Это помогает не допустить неправильного сочетания категории, должности и службы.

## Параметризованные SQL-запросы

При работе с базой данных используются параметризованные запросы через `SqlParameter`. Это реализовано в классе `DatabaseService`.

Параметры используются при:

- создании пользователя;
- проверке логина и пароля;
- создании кандидата;
- сохранении образования;
- создании карточки кандидата;
- сохранении этапов изучения;
- сохранении служебных проверок;
- сохранении итогового решения.

Такой подход снижает риск SQL-инъекций и помогает безопаснее передавать данные пользователя в SQL-запросы.

## Транзакция при создании карточки

Создание карточки кандидата выполняется в транзакции. В одной операции сохраняются:

- данные кандидата;
- сведения об образовании;
- карточка оформления.

Если на одном из этапов возникает ошибка, транзакция отменяется. Это защищает базу от частично сохраненных данных, например когда кандидат добавился, а карточка оформления не была создана.

## Ограничения на уровне базы данных

В базе данных используются первичные ключи, внешние ключи, уникальные ограничения и проверки `CHECK`.

Примеры реализованных ограничений:

- карточка оформления связана с существующим кандидатом;
- образование связано с кандидатом;
- этапы изучения связаны с карточкой кандидата;
- служебные проверки связаны с карточкой кандидата;
- решение связано с карточкой кандидата;
- для решения `Отказ` требуется причина;
- для решения `Принят` требуются реквизиты приказа.

Ограничения на уровне базы данных дополнительно защищают систему от некорректных записей.

## Триггеры и журналирование

Для серверной части подготовлены триггеры:

- `trg_Decisions_UpdateProcessingCardStage`;
- `trg_ProcessingCards_LogStageChange`.

Первый триггер автоматически обновляет статус карточки кандидата после сохранения итогового решения. Второй триггер записывает изменение статуса карточки в таблицу `CardChangeLog`.

Журналирование позволяет отследить, когда была создана карточка или изменен ее статус.

## Защита отчетов

Excel-отчет формируется автоматически на основе данных из приложения. Пользователь выбирает месяц и год, после чего приложение создает файл отчета.

Так как отчет может содержать персональные данные кандидатов, его необходимо сохранять в защищенной папке и передавать только уполномоченным сотрудникам. В рамках приложения реализована автоматическая генерация файла, что снижает риск ошибок ручного заполнения.

## Вывод

В проекте были реализованы базовые механизмы защиты: авторизация пользователей, хэширование паролей, проверка подключения к базе данных, проверка вводимых данных, ограничение возраста кандидата, использование справочников, параметризованные SQL-запросы, транзакции, ограничения целостности базы данных, триггеры и журналирование изменения статуса карточки.

Реализованные механизмы повышают надежность системы и уменьшают риск некорректного ввода, случайного изменения данных и несанкционированного доступа к приложению.
