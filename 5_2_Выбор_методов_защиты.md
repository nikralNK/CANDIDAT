# 5.2 Выбор методов защиты

После анализа существующих мер защиты необходимо выбрать методы, которые подходят для разрабатываемой информационной системы учета кандидатов. Так как система работает с персональными данными и кадровой информацией, защита должна быть направлена на ограничение доступа, контроль корректности данных и предотвращение случайного или несанкционированного изменения сведений.

Для проекта выбраны организационные, программные и серверные методы защиты.

## Разграничение доступа пользователей

Одним из основных методов защиты является разграничение доступа. В системе предполагаются две основные роли:

- кадровик;
- администратор.

Кадровик работает с карточками кандидатов: создает записи, заполняет сведения, ведет этапы изучения, служебные проверки и итоговые решения. Администратор должен отвечать за пользователей, справочники и настройки системы.

Разграничение прав необходимо для того, чтобы обычный пользователь не мог изменять служебные справочники или управлять учетными записями. Такой подход уменьшает риск случайного изменения важных данных.

## Авторизация пользователей

Для дальнейшего развития системы целесообразно реализовать авторизацию пользователей. Пользователь должен входить в приложение под своим логином и паролем. После входа система определяет его роль и показывает только доступные функции.

Авторизация нужна для того, чтобы:

- ограничить доступ к приложению;
- определить, кто работает с системой;
- связать действия пользователя с его учетной записью;
- разделить функции кадровика и администратора.

В учебной версии проекта таблица `Users` уже предусмотрена в базе данных, поэтому механизм авторизации можно добавить при дальнейшем развитии приложения.

## Использование параметризованных SQL-запросов

Для защиты от некорректного выполнения SQL-запросов выбран метод параметризации. В приложении значения передаются в запросы через `SqlParameter`, а не вставляются напрямую в текст SQL-запроса.

Это позволяет снизить риск SQL-инъекций и ошибок при передаче строковых данных. Такой способ используется при сохранении кандидата, образования, карточки оформления, этапов изучения, служебных проверок и решений.

## Проверка вводимых данных

Для защиты от некорректных данных используется проверка формы перед сохранением. Обязательные поля не должны быть пустыми, а дата рождения должна соответствовать требованию возраста кандидата.

Выбранные проверки:

- проверка фамилии, имени и отчества;
- проверка даты рождения;
- ограничение возраста кандидата 18+;
- проверка сведений об образовании;
- проверка выбранного источника сведений;
- проверка выбранной должности и службы;
- проверка даты заявления;
- проверка причины отказа при решении `Отказ`;
- проверка реквизитов приказа при решении `Принят`.

Проверка данных помогает уменьшить количество ошибок при вводе и не допустить сохранение неполной карточки.

## Использование справочников

Для защиты от ошибок ручного ввода используются справочники. Значения выбираются из выпадающих списков, а не вводятся вручную.

В системе используются справочники:

- источников сведений;
- категорий службы;
- должностей;
- служб;
- типов этапов изучения;
- типов служебных проверок.

Использование справочников позволяет сделать данные единообразными. Например, кадровик выбирает категорию службы, после чего приложение показывает только подходящие должности. Это снижает риск неправильного сочетания категории, должности и службы.

## Ограничения целостности в базе данных

Для защиты данных на уровне базы выбран метод использования ограничений целостности. В базе данных применяются:

- первичные ключи;
- внешние ключи;
- уникальные ограничения;
- ограничения `CHECK`;
- каскадное удаление для зависимых записей.

Эти ограничения помогают не допустить появления записей, которые не связаны с основными таблицами. Например, этап изучения не может существовать без карточки кандидата, а решение не может быть создано для несуществующей карточки.

## Хранимые процедуры

Для важных операций выбран метод использования хранимых процедур. Хранимые процедуры позволяют выполнять действия с данными централизованно на стороне MS SQL Server.

В проекте предусмотрены процедуры:

- создание карточки кандидата;
- сохранение этапа изучения;
- сохранение служебной проверки;
- сохранение итогового решения.

Процедура создания карточки выполняется в транзакции. Это защищает данные от частичного сохранения: если одна операция завершится ошибкой, все изменения отменяются.

## Триггеры

Для автоматического контроля данных выбраны триггеры. Они выполняются на стороне базы данных при изменении таблиц.

В проекте предусмотрены следующие триггеры:

- автоматическое обновление статуса карточки после изменения итогового решения;
- запись изменения статуса карточки в журнал.

Триггеры помогают поддерживать согласованность данных и фиксировать важные изменения.

## Журналирование действий

Для контроля изменения статусов выбран метод журналирования. В таблицу `CardChangeLog` записываются изменения стадии карточки кандидата.

Журналирование позволяет определить:

- у какой карточки изменился статус;
- какой статус был раньше;
- какой статус стал после изменения;
- когда произошло изменение.

В дальнейшем журналирование можно расширить и фиксировать не только изменение статуса, но и действия пользователей.

## Резервное копирование базы данных

Для защиты от потери данных необходимо использовать резервное копирование базы данных. В рамках проекта это можно рассматривать как организационно-техническую меру.

Резервное копирование позволяет восстановить данные в случае:

- случайного удаления информации;
- повреждения базы данных;
- ошибки пользователя;
- сбоя компьютера или SQL Server.

Для учебного проекта достаточно указать необходимость регулярного создания резервных копий средствами SQL Server.

## Защита Excel-отчетов

Excel-отчет содержит сведения о кандидатах, поэтому после формирования он также требует защиты. Необходимо ограничить доступ к папке, где сохраняются отчеты, и передавать такие файлы только уполномоченным сотрудникам.

Также рекомендуется использовать понятные имена файлов и хранить отчеты в отдельной защищенной папке.

## Вывод

Для разрабатываемой системы выбраны следующие методы защиты: разграничение ролей пользователей, авторизация, параметризованные SQL-запросы, проверка вводимых данных, использование справочников, ограничения целостности базы данных, хранимые процедуры, триггеры, журналирование и резервное копирование.

Выбранные методы позволяют повысить надежность системы, снизить риск ошибок при вводе данных и защитить сведения кандидатов от несанкционированного доступа и некорректного изменения.
