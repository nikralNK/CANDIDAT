# IDEF0. Функциональная модель предметной области

## A-0. Контекстная диаграмма

**Функция:** учет кандидатов на службу в Информационном центре МВД.

**Цель модели:** показать, как в системе происходит обработка сведений о кандидате: от получения данных до формирования итогового Excel-отчета.

**Точка зрения:** кадровик Информационного центра МВД.

### Входы

- Сведения кандидата.
- Документы и данные об образовании.
- Информация об источнике сведений.

### Управление

- Требования кадрового учета МВД.
- Правила обработки персональных данных.
- Шаблон ежемесячного отчета.
- Регламент изучения кандидата.

### Механизмы

- Кадровик.
- WPF-приложение.
- База данных MS SQL Server.
- Справочники системы.

### Выходы

- Карточка кандидата.
- Записи об этапах изучения.
- Записи о служебных проверках.
- Итоговое решение по кандидату.
- Excel-отчет по кандидатам.

## A0. Декомпозиция процесса учета кандидатов

На диаграмме A0 общий процесс разделяется на основные функции, которые выполняются в приложении.

### A1. Принять и проверить сведения кандидата

Кадровик получает сведения кандидата и проверяет, хватает ли данных для создания карточки. На этом этапе учитываются личные данные, дата рождения, образование и источник сведений.

**Входы:** сведения кандидата, документы, данные об образовании.  
**Управление:** требования кадрового учета, правила обработки персональных данных.  
**Механизмы:** кадровик, WPF-приложение.  
**Выходы:** проверенные сведения кандидата.

### A2. Создать карточку кандидата

Кадровик заносит сведения в приложение. Приложение сохраняет данные кандидата, образование, источник сведений, категорию службы, должность и наименование службы в базу данных.

**Входы:** проверенные сведения кандидата.  
**Управление:** справочники категорий, должностей и служб.  
**Механизмы:** WPF-приложение, MS SQL Server.  
**Выходы:** карточка кандидата.

### A3. Вести этапы изучения кандидата

Кадровик фиксирует этапы изучения кандидата: тип этапа, дату направления, дату результата и результат.

**Входы:** карточка кандидата.  
**Управление:** регламент изучения кандидата.  
**Механизмы:** кадровик, WPF-приложение, база данных.  
**Выходы:** записи об этапах изучения.

### A4. Внести служебные проверки

Кадровик заносит данные о служебных проверках кандидата: тип проверки, даты и результат.

**Входы:** карточка кандидата.  
**Управление:** регламент служебных проверок.  
**Механизмы:** кадровик, WPF-приложение, база данных.  
**Выходы:** записи о служебных проверках.

### A5. Принять решение и сформировать отчет

После заполнения данных кадровик выбирает итоговое решение: кандидат находится на оформлении, принят или получил отказ. После этого приложение формирует Excel-отчет за выбранный месяц.

**Входы:** карточка кандидата, этапы изучения, служебные проверки.  
**Управление:** шаблон отчета, правила принятия решения.  
**Механизмы:** кадровик, WPF-приложение, MS SQL Server.  
**Выходы:** итоговое решение, Excel-отчет.

## Связи между функциями A0

1. A1 передает проверенные сведения кандидата в A2.
2. A2 создает карточку кандидата, которая используется в A3 и A4.
3. A3 формирует записи об этапах изучения.
4. A4 формирует записи о служебных проверках.
5. A5 использует карточку, этапы и проверки для принятия решения и формирования отчета.
