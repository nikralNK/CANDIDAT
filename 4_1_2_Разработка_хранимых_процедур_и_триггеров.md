# 4.1.2 Разработка хранимых процедур и триггеров

Для серверной части информационной системы учета кандидатов были разработаны хранимые процедуры и триггеры. Они используются для переноса части логики обработки данных на уровень базы данных MS SQL Server.

Использование хранимых процедур позволяет централизовать основные операции с данными, уменьшить дублирование SQL-запросов в приложении и обеспечить более строгий порядок выполнения действий. Триггеры применяются для автоматического выполнения дополнительных операций при изменении данных.

## Хранимые процедуры

### `usp_CreateProcessingCard`

Процедура предназначена для создания новой карточки кандидата. Она выполняет сразу несколько действий:

- добавляет запись о кандидате в таблицу `Candidates`;
- добавляет сведения об образовании в таблицу `Education`;
- создает карточку оформления в таблице `ProcessingCards`;
- устанавливает начальную стадию `На оформлении`.

Процедура выполняется внутри транзакции. Это нужно для того, чтобы при ошибке не появилась ситуация, когда кандидат создан, а карточка оформления или образование не сохранены. Если ошибка возникает на любом этапе, все изменения отменяются.

### `usp_SaveStudyStage`

Процедура предназначена для сохранения этапа изучения кандидата. Если указанный этап уже существует у выбранной карточки, данные обновляются. Если такого этапа еще нет, создается новая запись.

Такой подход удобен для кадровика: он может повторно открыть этап и исправить дату или результат, не создавая дубликаты.

### `usp_SaveServiceCheck`

Процедура предназначена для сохранения служебной проверки кандидата. Логика аналогична этапам изучения: существующая проверка обновляется, а новая проверка добавляется.

Это позволяет хранить по одной записи каждого типа проверки для выбранной карточки кандидата.

### `usp_SaveDecision`

Процедура предназначена для сохранения итогового решения по кандидату. Она проверяет корректность данных:

- если выбран отказ, должна быть указана причина отказа;
- если кандидат принят, должны быть указаны реквизиты приказа.

Если решение по карточке уже существует, процедура обновляет его. Если решения еще нет, создается новая запись.

## Триггеры

### `trg_Decisions_UpdateProcessingCardStage`

Триггер срабатывает после добавления или изменения записи в таблице `Decisions`. Он автоматически обновляет поле `StudyStage` в таблице `ProcessingCards`.

Например:

- решение `Принят` переводит карточку в статус `Принят на службу`;
- решение `Отказ` переводит карточку в статус `Отказ`;
- решение `На оформлении` оставляет карточку в статусе `На оформлении`.

Благодаря этому статус карточки всегда соответствует последнему итоговому решению.

### `trg_ProcessingCards_LogStageChange`

Триггер срабатывает после создания или изменения карточки кандидата. Он записывает изменение стадии в таблицу `CardChangeLog`.

В журнал сохраняются:

- идентификатор карточки;
- тип действия;
- предыдущая стадия;
- новая стадия;
- дата и время изменения.

Журнал нужен для контроля изменений и может использоваться как простой механизм аудита.

## Таблица журнала изменений

Для работы триггера была добавлена таблица `CardChangeLog`. Она хранит историю создания карточек и изменения стадии оформления.

Поля таблицы:

- `LogId` — первичный ключ записи журнала;
- `CardId` — идентификатор карточки кандидата;
- `ActionName` — название действия;
- `OldStage` — старая стадия;
- `NewStage` — новая стадия;
- `ChangedAt` — дата и время изменения.

## Вывод

Разработанные хранимые процедуры и триггеры позволяют перенести часть обработки данных на уровень базы данных. Процедуры отвечают за создание карточки кандидата, сохранение этапов, проверок и итогового решения. Триггеры автоматически обновляют статус карточки и фиксируют изменения в журнале.

Такой подход повышает надежность работы системы, так как важные действия выполняются централизованно и контролируются средствами MS SQL Server.
