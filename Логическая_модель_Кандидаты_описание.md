# Логическая модель данных

## 1. Назначение логической модели

Логическая модель описывает основные сущности предметной области, их атрибуты и связи между ними. Она показывает, какие данные нужны системе учета кандидатов, но не привязывается к конкретным типам данных PostgreSQL или другой СУБД.

Модель построена под проект веб-приложения, в котором:

- кандидат заполняет анкету;
- кадровик ведет карточку оформления;
- кадровик фиксирует этапы изучения и служебные проверки;
- система формирует Excel-отчет по структуре исходного файла.

## 3. Основные сущности

| Сущность | Назначение |
|---|---|
| Пользователь | Хранит учетные записи кандидатов, кадровиков и администраторов |
| Кандидат | Хранит персональные данные кандидата |
| Образование | Хранит сведения об образовании кандидата |
| Заявление / карточка оформления | Центральная сущность, связывающая кандидата, подразделение, должность, статус и отчетные данные |
| Подразделение | Хранит подразделения МВД и территориальные органы |
| Должность | Хранит должности, службы, категории и состав |
| Этап изучения | Хранит этапы оформления: задание, УП/УР, испытания, ВВК, ППО |
| Служебная проверка | Хранит проверки ИЦ, ГИАЦ, ОРЧ СБ, ФСБ, ГИБДД и другие |
| Итоговое решение | Хранит результат оформления: принят, отказ, на оформлении |
| Выгрузка отчета | Хранит сведения о сформированных Excel-отчетах |

## 4. Атрибуты сущностей

### 4.1. Пользователь

| Атрибут | Описание |
|---|---|
| ID пользователя | Уникальный идентификатор пользователя |
| Роль | Кандидат, кадровик или администратор |
| Логин | Имя для входа в систему |
| Пароль | Данные для авторизации |
| ФИО | ФИО пользователя системы |
| Активность | Признак действующей учетной записи |

### 4.2. Кандидат

| Атрибут | Описание |
|---|---|
| ID кандидата | Уникальный идентификатор кандидата |
| ФИО | Фамилия, имя, отчество кандидата |
| Дата рождения | Дата рождения кандидата |

| Атрибут | Описание |

## 5. Связи и кардинальности

| Связь | Кардинальность | Описание |
|---|---|---|
| Пользователь - Кандидат | 1 : 0..1 | Один пользователь может быть связан с одной анкетой кандидата |
| Кандидат - Образование | 1 : 0..N | У кандидата может быть несколько записей об образовании |
| Кандидат - Заявление | 1 : 0..N | Один кандидат может иметь несколько заявлений |
| Подразделение - Заявление | 1 : 0..N | В одно подразделение могут поступать разные заявления |
| Должность - Заявление | 1 : 0..N | На одну должность могут быть разные заявления |
| Заявление - Этап изучения | 1 : 0..N | У заявления может быть несколько этапов оформления |
| Заявление - Служебная проверка | 1 : 0..N | У заявления может быть несколько служебных проверок |
| Заявление - Итоговое решение | 1 : 0..1 | У заявления может быть одно итоговое решение |
| Пользователь - Выгрузка отчета | 1 : 0..N | Один пользователь может сформировать несколько отчетов |

## 6. Правила предметной области

1. Кандидат может сохранить анкету как черновик.
2. После отправки анкеты создается заявление.
3. Кадровик может принять заявление в работу и вести карточку оформления.
4. Служебные проверки заполняет кадровик, а не кандидат.
5. Итоговое решение может быть только одно для одного заявления.
6. Если кандидат принят, должны быть указаны реквизиты приказа.
7. Если кандидату отказано, должны быть указаны дата и причина отказа.
8. Excel-отчет формируется на основании заявления, данных кандидата, этапов, проверок и решения.

## 7. Отличие логической модели от физической

| Логическая модель | Физическая модель |
|---|---|
| Описывает сущности предметной области | Описывает таблицы базы данных |
| Использует понятные названия атрибутов | Использует SQL-поля и типы данных |
| Показывает кардинальности связей | Показывает PK, FK, CHECK, INDEX |
| Не зависит от конкретной СУБД | Может быть реализована в PostgreSQL |

## 9. Вывод

Логическая модель подходит для проекта, потому что она отражает реальный процесс работы с кандидатом: сначала кандидат заполняет анкету, затем кадровик ведет оформление и проверки, после чего система формирует Excel-отчет. Модель является основой для физической модели базы данных и дальнейшей разработки приложения.
