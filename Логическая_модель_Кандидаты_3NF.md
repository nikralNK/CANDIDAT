# Логическая модель данных в 3НФ

## 1. Что означает 3НФ

Третья нормальная форма означает, что:

1. В таблицах нет повторяющихся групп данных.
2. Каждый неключевой атрибут зависит от ключа.
3. Неключевые атрибуты не зависят друг от друга.
4. Справочные данные вынесены в отдельные сущности.

Для проекта это важно, потому что кандидат выбирает значения из справочников, кадровик ведет оформление, а Excel-отчет автоматически собирается из связанных сущностей без ручного дублирования данных.

## 2. Основные изменения для 3НФ

| Было рискованно | Как сделано в 3НФ |
|---|---|
| Категория, служба, подразделение и должность могли храниться текстом в анкете | В анкете хранится ссылка на `Вакансия / доступная должность` |
| Данные должности могли повторяться в анкете и карточке оформления | В карточке оформления хранится ссылка на анкету, а выбранная должность берется через анкету |
| Настройки колонок Excel могли храниться одним текстовым полем в шаблоне | Колонки отчета вынесены в отдельную сущность `Колонка шаблона отчета` |
| Источники сведений, категории, службы и должности могли быть текстом | Все вынесено в отдельные справочники |

## 3. Сущности 3НФ

| Сущность | Назначение |
|---|---|
| Пользователь | Учетные записи пользователей |
| Кандидат | Персональные данные кандидата |
| Анкета кандидата | Анкета, которую кандидат заполняет и отправляет кадровику |
| Образование | Сведения об образовании кандидата |
| Источник сведений | Справочник источников информации о трудоустройстве |
| Категория службы | Справочник категорий: юстиция, полиция, внутренняя служба |
| Подразделение | Справочник подразделений |
| Служба | Справочник служб |
| Должность | Справочник должностей |
| Вакансия / доступная должность | Связь категории, службы, подразделения и должности |
| Карточка оформления | Рабочая карточка кадровика по принятой анкете |
| Тип этапа изучения | Справочник типов этапов |
| Этап изучения | Даты и результаты этапов оформления |
| Тип служебной проверки | Справочник типов проверок |

## 4. Атрибуты сущностей

### 4.1. Пользователь

| Атрибут | Описание |
|---|---|
| ID пользователя | Уникальный идентификатор |
| ФИО пользователя | ФИО владельца учетной записи |
| Логин | Логин для входа |
| Пароль | Данные авторизации |
| Роль | Кандидат, кадровик, администратор |
| Статус учетной записи | Активен или заблокирован |

### 4.2. Кандидат

| Атрибут | Описание |
|---|---|
| ID кандидата | Уникальный идентификатор |
| ID пользователя | Связь с учетной записью |
| ФИО кандидата | ФИО кандидата |
| Дата рождения | Дата рождения кандидата |
| Телефон | Контактный телефон |
| Email | Электронная почта |

### 4.3. Анкета кандидата

| Атрибут | Описание |
|---|---|
| ID анкеты | Уникальный идентификатор анкеты |
| ID кандидата | Кто заполнил анкету |
| ID источника сведений | Откуда кандидат узнал о работе |
| ID вакансии | Выбранная связка категории, службы, подразделения и должности |
| Дата создания | Когда анкета создана |
| Дата отправки | Когда анкета отправлена кадровику |
| Статус анкеты | Черновик, отправлена, возвращена, принята |
| Комментарий кадровика | Причина возврата на доработку |
| Иные сведения | Дополнительная информация |

Важный момент 3НФ: в анкете не хранятся отдельно `категория`, `служба`, `подразделение` и `должность`. Хранится только `ID вакансии`, а все официальные значения берутся через связанные справочники.

### 4.10. Вакансия / доступная должность

Эта сущность связывает справочники и позволяет системе фильтровать выпадающие списки.

| Атрибут | Описание |
|---|---|
| ID вакансии | Уникальный идентификатор |
| ID категории | Категория службы |
| ID подразделения | Подразделение |
| ID службы | Служба |
| ID должности | Должность |
| Активность | Можно ли выбрать эту связку |

Пример:

| Категория | Подразделение | Служба | Должность |
|---|---|---|---|
| Юстиция | ОМВД России по району | СО | Следователь |
| Полиция | ОМВД России по району | УУП | Участковый уполномоченный полиции |

### 4.11. Карточка оформления

| Атрибут | Описание |
|---|---|
| ID карточки | Уникальный идентификатор |

## 6. Почему модель соответствует 3НФ

| Правило 3НФ | Как выполнено |
|---|---|
| Нет повторяющихся групп | Этапы, проверки, образования и колонки отчета вынесены в отдельные сущности |
| Нет хранения справочных текстов в основных сущностях | Анкета хранит ID источника и ID вакансии, а не текстовые значения |
| Нет транзитивных зависимостей | Категория, служба, подразделение и должность не зависят от анкеты напрямую, они зависят от вакансии |
| Нет дублирования выбранной должности | Карточка оформления не хранит должность повторно, она получает ее через анкету |
| Настройки Excel нормализованы | Колонки шаблона вынесены отдельно от шаблона |

## 7. Как данные попадут в Excel

| Колонка Excel | Источник в модели 3НФ |
|---|---|
| ФИО кандидата | Кандидат |
| дата рождения | Кандидат |
| образование | Образование |
| источник сведений | Источник сведений через Анкету |
| категория | Категория службы через Вакансию |
| подразделение | Подразделение через Вакансию |
| служба | Служба через Вакансию |
| должность | Должность через Вакансию |
| начальствующий состав | Должность |
| этапы изучения | Этап изучения + Тип этапа изучения |
| служебные проверки | Служебная проверка + Тип служебной проверки |
| отказ / приказ | Итоговое решение |
| готовый Excel-файл | Выгрузка отчета + Шаблон отчета + Колонки шаблона отчета |

## 8. Вывод

Эта логическая модель приведена к третьей нормальной форме. Она подходит для автоматизации, потому что данные не дублируются, справочники управляются отдельно, кандидат выбирает только допустимые значения, а Excel-отчет автоматически формируется из связанных сущностей.
