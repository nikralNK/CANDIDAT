# Физическая модель базы данных

## 1. Назначение физической модели

Физическая модель описывает, как данные веб-приложения будут храниться в базе данных. Модель сделана под проект учета кандидатов для приема на службу и формирования ежемесячного Excel-отчета по исходному шаблону.

Главная идея: кандидат заполняет только анкетные данные, а кадровик отдельно ведет служебные этапы, проверки и итоговое решение. Поэтому данные не хранятся в одной большой таблице, а разделяются на связанные таблицы.

## 2. СУБД

Рекомендуемая СУБД: `PostgreSQL`.

Для учебного проекта также можно использовать `SQLite`, но PostgreSQL лучше подходит для полноценного веб-приложения, потому что поддерживает роли, ограничения, индексы и надежную работу с несколькими пользователями.

## 4. Таблицы физической модели

### 4.1. users

Таблица хранит учетные записи пользователей системы.

| Поле | Тип | Назначение |
|---|---|---|
| id | BIGSERIAL PK | Уникальный идентификатор пользователя |
| role | VARCHAR(30) | Роль: кандидат, кадровик, администратор |
| login | VARCHAR(100) UNIQUE | Логин пользователя |
| password_hash | VARCHAR(255) | Хеш пароля |
| full_name | VARCHAR(255) | ФИО пользователя системы |
| is_active | BOOLEAN | Признак активности учетной записи |
| created_at | TIMESTAMP | Дата создания |

### 4.2. candidates

Таблица хранит персональные данные кандидата.

| Поле | Тип | Назначение |
|---|---|---|
| id | BIGSERIAL PK | Уникальный идентификатор кандидата |
| user_id | BIGINT FK | Связь с учетной записью кандидата |
| full_name | VARCHAR(255) | ФИО кандидата |
| birth_date | DATE | Дата рождения |
| phone | VARCHAR(30) | Телефон |
| email | VARCHAR(150) | Email |
| address | TEXT | Адрес проживания |
| created_at | TIMESTAMP | Дата создания записи |
| updated_at | TIMESTAMP | Дата последнего изменения |

Соответствие Excel: `ФИО кандидата`, `дата рождения кандидата`.

### 4.3. educations

Таблица хранит сведения об образовании кандидата.

| Поле | Тип | Назначение |
|---|---|---|
| id | BIGSERIAL PK | Уникальный идентификатор записи |
| candidate_id | BIGINT FK | Связь с кандидатом |
| education_level | VARCHAR(100) | Уровень образования |
| education_type | VARCHAR(150) | Вид образования |
| institution | VARCHAR(255) | Учебное заведение |
| document_number | VARCHAR(100) | Номер документа об образовании |
| graduation_year | INTEGER | Год окончания |
| created_at | TIMESTAMP | Дата создания |

Соответствие Excel: `уровень и вид образования кандидата`.

### 4.4. departments

Таблица хранит подразделения МВД и территориальные органы.

| Поле | Тип | Назначение |
|---|---|---|
| id | BIGSERIAL PK | Уникальный идентификатор подразделения |
| name | VARCHAR(255) | Наименование подразделения |
| department_type | VARCHAR(100) | Тип подразделения |
| parent_id | BIGINT NULL | Родительское подразделение |
| is_active | BOOLEAN | Используется ли подразделение |
| created_at | TIMESTAMP | Дата создания |

Соответствие Excel: `подразделение МВД, территориальный орган МВД`.

### 4.5. positions

Таблица хранит должности, службы, категории и уровень начальствующего состава.

| Поле | Тип | Назначение |
|---|---|---|
| id | BIGSERIAL PK | Уникальный идентификатор должности |
| name | VARCHAR(255) | Наименование должности |
| service_name | VARCHAR(255) | Наименование службы |
| command_staff_level | VARCHAR(100) | Начальствующий состав |
| category | VARCHAR(100) | Категория: полиция, юстиция, внутренняя служба |
| is_active | BOOLEAN | Используется ли должность |
| created_at | TIMESTAMP | Дата создания |
| updated_at | TIMESTAMP | Дата изменения |

Соответствие Excel: `начальствующий состав`, `категория`, `наименование должности`, `наименование службы`.

### 4.6. applications

Главная таблица заявления и карточки оформления кандидата.

| Поле | Тип | Назначение |
|---|---|---|
| id | BIGSERIAL PK | Уникальный идентификатор заявления |
| candidate_id | BIGINT FK | Связь с кандидатом |
| department_id | BIGINT FK | Подразделение |
| position_id | BIGINT FK | Должность |
| statement_date | DATE | Дата заявления |
| status | VARCHAR(50) | Статус заявления |
| employment_source | TEXT | Источник получения сведений о трудоустройстве |
| other_info | TEXT | Иные сведения |
| submitted_at | TIMESTAMP | Дата отправки заявления |
| created_at | TIMESTAMP | Дата создания |
| updated_at | TIMESTAMP | Дата изменения |

Допустимые значения `status`:

| Значение | Смысл |
|---|---|
| draft | Черновик |
| submitted | Отправлено кадровику |
| in_progress | На оформлении |
| accepted | Принят на службу |
| rejected | Отказ |

Соответствие Excel: `дата заявления`, `стадия изучения`, `источник получения сведений`, `иные сведения`.

### 4.7. recruitment_stages

Таблица хранит этапы изучения кандидата.

| Поле | Тип | Назначение |
|---|---|---|
| id | BIGSERIAL PK | Уникальный идентификатор этапа |
| application_id | BIGINT FK | Связь с заявлением |
| stage_type | VARCHAR(80) | Тип этапа |
| sent_date | DATE | Дата направления или утверждения |
| result_date | DATE | Дата результата |
| result_text | TEXT | Результат |
| comment | TEXT | Комментарий |
| created_at | TIMESTAMP | Дата создания |
| updated_at | TIMESTAMP | Дата изменения |

Типы этапов:

| stage_type | Колонка Excel |
|---|---|
| TASK_APPROVAL | дата утверждения задания для изучения кандидата |
| UP_UR_DIRECTION | дата направления установок УП/УР |
| TESTING | дата направления кандидата на испытания / результат |
| VVK | дата направления на ВВК, дата прохождения и результат ВВК |
| PPO | дата и результат ППО |

### 4.8. candidate_checks

Таблица хранит служебные проверки кандидата.

| Поле | Тип | Назначение |
|---|---|---|

| sent_date | DATE | Дата направления проверки |

| result_text | TEXT | Результат проверки |

Типы проверок:

| GIAC | дата направления проверки ГИАЦ |
| ORCH_SB | дата направления согласования ОРЧ СБ |
| FSB | дата направления материалов в ФСБ |
| GIBDD | дата проверки ГИБДД |
| BSTM | дата направления проверки БСТМ |

### 4.9. decisions

Таблица хранит итоговое решение по кандидату.

| Поле | Тип | Назначение |
|---|---|---|
| id | BIGSERIAL PK | Уникальный идентификатор решения |
| application_id | BIGINT FK UNIQUE | Связь с заявлением |
| decision_type | VARCHAR(30) | Тип решения |
| decision_date | DATE | Дата решения |
| refusal_reason | TEXT | Причина отказа |
| order_details | TEXT | Реквизиты приказа о приеме / направление в УРЛС |
| created_at | TIMESTAMP | Дата создания |

Допустимые значения `decision_type`:

| Значение | Смысл |
|---|---|
| accepted | Принят на службу |
| rejected | Отказ |
| in_progress | На оформлении |

Соответствие Excel: `дата и причина отказа`, `реквизиты приказа о приеме / дата направления в УРЛС`.

### 4.10. report_exports

Таблица хранит историю сформированных отчетов.

| Поле | Тип | Назначение |
|---|---|---|
| id | BIGSERIAL PK | Уникальный идентификатор выгрузки |
| created_by | BIGINT FK | Пользователь, сформировавший отчет |
| period_month | INTEGER | Месяц отчета |
| period_year | INTEGER | Год отчета |
| file_name | VARCHAR(255) | Имя Excel-файла |
| file_path | VARCHAR(500) | Путь к файлу |
| created_at | TIMESTAMP | Дата формирования |

## 5. Связи между таблицами

| Связь | Тип | Описание |
|---|---|---|
| users -> candidates | 1 к 0..1 | У кандидата может быть учетная запись |
| candidates -> educations | 1 ко многим | У кандидата может быть несколько образований |
| candidates -> applications | 1 ко многим | Кандидат может подать несколько заявлений |
| departments -> applications | 1 ко многим | В подразделение поступают заявления |
| positions -> applications | 1 ко многим | На должность могут претендовать разные кандидаты |
| applications -> recruitment_stages | 1 ко многим | В заявлении много этапов изучения |
| applications -> candidate_checks | 1 ко многим | В заявлении много служебных проверок |
| applications -> decisions | 1 к 0..1 | По заявлению может быть одно итоговое решение |
| users -> report_exports | 1 ко многим | Пользователь может сформировать много отчетов |

## 6. SQL-скрипт создания таблиц

```sql
CREATE TABLE users (
    id              BIGSERIAL PRIMARY KEY,
    role            VARCHAR(30) NOT NULL CHECK (role IN ('candidate', 'hr', 'admin')),
    login           VARCHAR(100) NOT NULL UNIQUE,
    password_hash   VARCHAR(255) NOT NULL,
    full_name       VARCHAR(255) NOT NULL,
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE candidates (
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT UNIQUE REFERENCES users(id),
    full_name   VARCHAR(255) NOT NULL,
    birth_date  DATE NOT NULL,
    phone       VARCHAR(30),
    email       VARCHAR(150),
    address     TEXT,
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE educations (
    id              BIGSERIAL PRIMARY KEY,
    candidate_id    BIGINT NOT NULL REFERENCES candidates(id) ON DELETE CASCADE,
    education_level VARCHAR(100) NOT NULL,
    education_type  VARCHAR(150),
    institution     VARCHAR(255),
    document_number VARCHAR(100),
    graduation_year INTEGER,
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE departments (
    id              BIGSERIAL PRIMARY KEY,
    name            VARCHAR(255) NOT NULL,
    department_type VARCHAR(100),
    parent_id       BIGINT REFERENCES departments(id),
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE positions (
    id                  BIGSERIAL PRIMARY KEY,
    name                VARCHAR(255) NOT NULL,
    service_name        VARCHAR(255),
    command_staff_level VARCHAR(100),
    category            VARCHAR(100),
    is_active           BOOLEAN NOT NULL DEFAULT TRUE,
    created_at          TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at          TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE applications (
    id                  BIGSERIAL PRIMARY KEY,
    candidate_id        BIGINT NOT NULL REFERENCES candidates(id),
    department_id       BIGINT REFERENCES departments(id),
    position_id         BIGINT REFERENCES positions(id),
    statement_date      DATE NOT NULL,
    status              VARCHAR(50) NOT NULL CHECK (status IN ('draft', 'submitted', 'in_progress', 'accepted', 'rejected')),
    employment_source   TEXT,
    other_info          TEXT,
    submitted_at        TIMESTAMP,
    created_at          TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at          TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE recruitment_stages (
    id              BIGSERIAL PRIMARY KEY,
    application_id  BIGINT NOT NULL REFERENCES applications(id) ON DELETE CASCADE,
    stage_type      VARCHAR(80) NOT NULL CHECK (stage_type IN ('TASK_APPROVAL', 'UP_UR_DIRECTION', 'TESTING', 'VVK', 'PPO')),
    sent_date       DATE,
    result_date     DATE,
    result_text     TEXT,
    comment         TEXT,
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE candidate_checks (
    id              BIGSERIAL PRIMARY KEY,
    application_id  BIGINT NOT NULL REFERENCES applications(id) ON DELETE CASCADE,
    check_type      VARCHAR(80) NOT NULL CHECK (check_type IN ('IC', 'GIAC', 'ORCH_SB', 'FSB', 'GIBDD', 'BSTM', 'SOOP', 'UVM', 'DIPLOMA', 'SGT_ACCESS')),
    sent_date       DATE,
    result_date     DATE,
    result_text     TEXT,
    comment         TEXT,
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE decisions (
    id              BIGSERIAL PRIMARY KEY,
    application_id  BIGINT NOT NULL UNIQUE REFERENCES applications(id) ON DELETE CASCADE,
    decision_type   VARCHAR(30) NOT NULL CHECK (decision_type IN ('accepted', 'rejected', 'in_progress')),
    decision_date   DATE,
    refusal_reason  TEXT,
    order_details   TEXT,
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE report_exports (
    id              BIGSERIAL PRIMARY KEY,
    created_by      BIGINT NOT NULL REFERENCES users(id),
    period_month    INTEGER NOT NULL CHECK (period_month BETWEEN 1 AND 12),
    period_year     INTEGER NOT NULL CHECK (period_year >= 2000),
    file_name       VARCHAR(255) NOT NULL,
    file_path       VARCHAR(500),
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## 7. Индексы

```sql
CREATE INDEX idx_candidates_full_name
    ON candidates(full_name);

CREATE INDEX idx_applications_candidate_id
    ON applications(candidate_id);

CREATE INDEX idx_applications_status
    ON applications(status);

CREATE INDEX idx_applications_statement_date
    ON applications(statement_date);

CREATE INDEX idx_recruitment_stages_application_id
    ON recruitment_stages(application_id);

CREATE INDEX idx_candidate_checks_application_id
    ON candidate_checks(application_id);

CREATE INDEX idx_report_exports_period
    ON report_exports(period_year, period_month);
```

## 8. Почему модель подходит для проекта

Эта физическая модель подходит для проекта по нескольким причинам:

1. Данные кандидата отделены от служебных данных кадровика.
2. Этапы изучения и проверки можно расширять без изменения основной таблицы заявлений.
3. По каждой заявке можно сформировать строку Excel-отчета.
4. Система поддерживает роли пользователей.
5. Есть история сформированных отчетов.
6. Структура БД соответствует исходному Excel-шаблону, но остается нормализованной.
