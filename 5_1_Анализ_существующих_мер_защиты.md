# 5.1 Анализ существующих мер защиты

Разрабатываемая информационная система предназначена для учета кандидатов на службу в Информационном центре МВД. В системе обрабатываются сведения, относящиеся к персональным данным: ФИО кандидата, дата рождения, контактные данные, сведения об образовании, источник получения сведений, должность, служба, этапы изучения, служебные проверки и итоговое решение.

Так как система работает с персональными данными, необходимо учитывать вопросы защиты информации. Основная задача защиты заключается в том, чтобы исключить несанкционированный доступ к данным, уменьшить риск случайного изменения или удаления сведений, а также обеспечить корректность вводимой информации.

## Характер обрабатываемой информации

В системе хранятся следующие виды данных:

- личные данные кандидата;
- контактная информация;
- сведения об образовании;
- данные о выбранной категории службы, должности и службе;
- сведения об этапах изучения кандидата;
- сведения о служебных проверках;
- итоговое решение по кандидату;
- данные пользователей системы.

Данная информация требует аккуратного обращения, так как она связана с кадровым учетом и может содержать персональные сведения. Поэтому доступ к ней должен быть ограничен только сотрудниками, которым эти данные необходимы для выполнения служебных обязанностей.

## Существующие меры защиты

На текущем этапе в проекте уже используются следующие меры защиты.

### Ограничение доступа к рабочему месту

Приложение является desktop-приложением и запускается на рабочем компьютере. Доступ к компьютеру ограничивается учетной записью Windows. Это является базовой мерой защиты, так как посторонний пользователь не должен иметь возможность открыть приложение без доступа к рабочему месту.

### Хранение данных в базе данных

Данные кандидатов хранятся не в разрозненных Excel-файлах, а в базе данных MS SQL Server. Такой подход позволяет централизованно хранить сведения, использовать связи между таблицами и уменьшить вероятность потери данных.

База данных содержит отдельные таблицы для кандидатов, образования, карточек оформления, этапов изучения, служебных проверок, решений и справочников. Это делает хранение данных более структурированным и управляемым.

### Использование справочников

В приложении используются справочники для выбора категории службы, должности, наименования службы, источника сведений, типов этапов и типов проверок. Это снижает риск ошибок при ручном вводе.

Например, кадровик не вводит должность и службу вручную, а выбирает их из выпадающих списков. После выбора категории службы приложение показывает только подходящие должности, а после выбора должности — связанные с ней службы.

### Проверка вводимых данных

Перед сохранением карточки кандидата приложение проверяет заполнение обязательных полей. Проверяются фамилия, имя, отчество, дата рождения, сведения об образовании, источник сведений, должность, служба и дата заявления.

Также реализована проверка возраста кандидата. Дата рождения ограничивается так, чтобы кандидату было не меньше 18 лет. Это предотвращает сохранение некорректных данных.

### Параметризованные SQL-запросы

При работе с базой данных используются параметризованные SQL-запросы через `SqlParameter`. Это позволяет не вставлять данные пользователя напрямую в текст SQL-запроса и снижает риск SQL-инъекций.

Параметризованные запросы используются при сохранении кандидата, образования, карточки оформления, этапов изучения, служебных проверок и решений.

### Ограничения на уровне базы данных

В базе данных используются первичные ключи, внешние ключи и ограничения. Они помогают контролировать целостность данных.

Примеры ограничений:

- карточка кандидата связана с существующим кандидатом;
- этапы изучения связаны с конкретной карточкой;
- служебные проверки связаны с конкретной карточкой;
- решение может быть связано только с существующей карточкой;
- для решения `Отказ` должна быть указана причина;
- для решения `Принят` должны быть указаны реквизиты приказа.

Такие ограничения помогают избежать некорректных записей в базе данных.

### Журналирование изменений

Для контроля изменения статуса карточки предусмотрена таблица `CardChangeLog` и триггер `trg_ProcessingCards_LogStageChange`. При создании карточки или изменении стадии оформления в журнал записывается идентификатор карточки, старый статус, новый статус и дата изменения.

Журналирование позволяет отследить, когда менялся статус карточки кандидата.

## Недостатки существующих мер защиты

Несмотря на наличие базовых мер защиты, текущая версия проекта является учебным прототипом и имеет ограничения.

К основным недостаткам можно отнести:

- в приложении пока нет полноценной авторизации пользователей;
- роли кадровика и администратора не разделены на уровне интерфейса;
- пароль пользователя в учебном примере не используется для входа;
- отсутствует подробный журнал всех действий пользователя;
- не реализовано резервное копирование базы данных средствами приложения;
- не настроено шифрование отдельных персональных данных;
- права доступа к таблицам базы данных не разграничены детально.

Эти ограничения можно устранить при дальнейшем развитии системы.

## Возможные угрозы

Для системы учета кандидатов можно выделить следующие возможные угрозы:

- несанкционированный доступ к компьютеру с установленным приложением;
- просмотр персональных данных посторонним пользователем;
- случайное изменение или удаление данных;
- ввод некорректных сведений;
- потеря данных при отсутствии резервного копирования;
- передача Excel-отчета неуполномоченным лицам;
- ошибки при ручном управлении справочниками.

Для уменьшения данных рисков необходимо развивать механизмы защиты системы.

## Вывод

В текущей версии проекта реализованы базовые меры защиты: хранение данных в MS SQL Server, использование справочников, проверка вводимых данных, параметризованные SQL-запросы, ограничения целостности на уровне базы данных и журналирование изменения статуса карточки.

Однако для полноценной эксплуатации системы необходимо дополнительно реализовать авторизацию пользователей, разграничение прав доступа, расширенное журналирование действий, резервное копирование и организационные меры защиты персональных данных.
